## Преобразование без unsafe

Возьмём задачу обработки массива байт и преобразование его в числа (для нас роли не играет,
но допустим, массивы байт будут содержать закодированные кодировкой [utf-8](https://en.wikipedia.org/wiki/UTF-8), целые, неотрицательные, 64-битные
числа в десятеричной системе счисления. Сами массивы байт мы получаем
из некоторого источника `src`.

Структура данных представляющая последовательность байт в языке Go называется [Array](https://go.dev/ref/spec#Array_types) и реализованный на базе `Array` тип [Slice](https://go.dev/ref/spec#Slice_types), `Slice` имеет чуть больше возможностей и концептуально не отличается от `Array`, будем использовать его. С терминологией определились, идём дальше. Алгоритм.

Наш алгортм прост. Выделяем на стеке (стекфрейме функции handle) место для промежуточного хранения вычитанной
из сокета порции данных. На каждой итерации цикла копируем из сокета данные на стек фнкции handle, обрабатываем их, проводя некие вычисления с полученным результатом. Наивное решение будет выглядеть как-нибудь так[^8]:

```go
package main

import (
	"errors"
	"fmt"
	"io"
	"log"
	"strconv"
)

func check(err error) {
	if err != nil {
		log.Fatal(err)
	}
}

type socket struct {
	eof bool
	src []byte
}

func (s *socket) ReadFrom(dst []byte) (int, error) {
	if s.eof {
		return 0, io.EOF
	}

	if len(s.src) > len(dst) {
		return 0, fmt.Errorf("dst buffer is too small, want=%d, got=%d", len(s.src), len(dst))
	}

	s.eof = true

	return copy(dst, s.src), nil
}

func handle(s *socket) uint64 {
	buf := [128]byte{}
	total := uint64(0)

	for {
		n, err := s.ReadFrom(buf[:])
		if errors.Is(err, io.EOF) {
			break
		}
		check(err)

		nn, err := strconv.ParseUint(buf[:n], 10, 64)
		check(err)

		total += nn
	}

	return total
}

func main() {
	s := socket{
		eof: false,
		src: []byte("9223372036854775807"),
	}

	fmt.Println(handle(&s))
}
```

Ага, этот код не компилируется. Всё дело в несовпадении типа аргумента `buf[:n]`
и типа параметра функции [ParseUint](https://pkg.go.dev/strconv#ParseUint),
byte-slice и [string](https://go.dev/ref/spec#String_types) соответственно.
Функции пакета `strconv` (и `ParseUint` в том числе) предназначены для обработки строк, но не слайсов байт (даже название намекает).

В Go есть механизм, позволяющий _изменить_ исходный тип данных на другой, механизм этот называется [Type Conversions](https://go.dev/ref/spec#Conversions). Синтаксически одинаков для всех пар типов, реализация своя (если она вообще есть). И необходимое нам правило есть в языке, так оно звучит:

> Converting a slice of bytes to a string type yields a string whose successive bytes are the elements of the slice.

То что нужно, воспользуемся им:

```diff
- v, err := strconv.ParseUint(buf[:n], 10, 64)
+ v, err := strconv.ParseUint(string(buf[:n]), 10, 64)
```

Всё работает, но есть проблема. Конверсия типов _может_ работать
неэффективно. Хотя правило не говорит о реализации (оставляя пространство для оптимизации в будущем, что правильно), мы вольны изучать реализацию и делать выводы. Самый простой способ это посмотреть изучив вывод инструмента `go tool objdump` с флагом `-S`. Нас интересует только момент вызова функции `strconv.ParseUint` и вычисление её аргументов:

```asm
CALL runtime.slicebytetostring
MOVL $0xa,  CX
MOVL $0x40, DI
CALL strconv.ParseUint
```

Учтём что строка есть струтура с двумя полями, указателем на первый байт строки и длина в байтах. Функция `strconv.ParseUint` принимает 3 агрумента. Строку, основание системы счисления и разрядность числа. `strconv.ParseUint` будет ждать в регистрах `AX/BX/CX/DI` указатель на первый байт строки, длину строки, основание системы счисления и разрядность числа соответственно.

Как мы видим, в регистры `CX` и `DI` помещаеюся числа `10` и `64`. А вот в регистры `AX` и `BX` компилятор предполагает поместить указатель на строку (на сами данные, не указатель на _заголокок строки_) и длину соответственно. Но для это сначала строку нужно получить, поэтому компилятор вставляет вызов функции `slicebytetostring`, которая и преобразует слайс байт в строку. Упрощённо её реализация выглядит так:

```go
func slicebytetostring(buf *tmpBuf, ptr *byte, n int) (str string) {
	var p unsafe.Pointer
	if buf != nil && n <= len(buf) {
		p = unsafe.Pointer(buf)
	} else {
		p = mallocgc(uintptr(n), nil, false)
	}
	stringStructOf(&str).str = p
	stringStructOf(&str).len = n
	memmove(p, unsafe.Pointer(ptr), uintptr(n))
	return
}
```

Сначала мы проверяем выдали ли нам буфер куда мы поместим содержание строки, если нет или длина буфера недостаточна мы аллоцируем в куче пространство для строки (да, современный компилятор не всегда может понять что объекту не нужно находиться в куче, даже если это реально так).

Наша задача не предполагает использоание старого значения слайса (число уже распаршено) после завершения очередной итерации цикла, и, так же не предполагает дальнейшее использование строки. Но мы всё равно будем конструировать строку, аллоцируя память, копировать содержимое слайса в место выделенное под строку. И так на каждой итерации. Мы можем нагружаем рантайм бесполезной работой, на работу аллокатора требуется CPU, на отслеживание выделенных объектов и их дальнейшее освобождение сборщиком мусора так же тратит ресурсы CPU[^1].

## Преобразование с unsafe

Как сказано выше, нам не нужна строка, мы лишь удовлетворяем требованиея типа параметра функции `ParseUint`.

Читатель может сказать что строки и слайсы байт похожи по структуре в памяти. И можно заставить компилятор интерпретировать некоторый участок памяти выделенный под слайс байт как со строкой. Обмануть и сказать работай как со строкой.

Тут нам поможет пакет unsafe и тот же самый механизм конверцсии типов.

Идея заключается в том, что мы можем сказать компилатору, на самом деле, в этом участке памяти лежит не слайс, там лежит строка. На помощь придет всё та же конверсия типов. Мы можем взять указатель на слайс, и сказать что это указатель на строку. А перед использованием строки, разименовть указатель.

```go
buf := []byte{'H', 'e', 'l', 'l'}
pointerToByteSlice := unsafe.Pointer(&buf)
pointerToString := (*string)pointer
str := *pointerToString
fmt.Printf("val=%q type=%T\n", str, str) // val="Hell" type=string
```

Это работает потому что представление слайса байт в памяти очень похоже на представление строк.

Представление слайса байт и строк:

```go
// runtime/slice.go
type slice struct {
	array unsafe.Pointer
	len   int
	cap   int
}

// runtime/string.go
type stringStruct struct {
	str unsafe.Pointer
	len int
}
```

Первые 8[^9] байт и там и там содержат указатель на сами данные. Компилятор не переупорядочивает в памяти хранимые поля структуры. Поэтому можно взять кусок памяти в котором хранится слайс и сказать, это строка.

Когда рантайм пойдёт брать длину строки, так так же отсчитает 8 байт от начала и прочитает 8 байт где найдёт длину строки. А когда возьмёт 8 байт начиная с 0, он обнажит адрес, по которому лежат байты нашей строки. Как мы видели в реализации функции `slicebytetostring` содержимое слайса байт не проходит никакой обрабокт (например проверки что последовательность это валидация последовательность udf-8 символов).

Однострочная версия версия:

```
buf := []byte{'H', 'e', 'l', 'l'}
v, err := strconv.ParseUint(*(*string)(unsafe.Pointer(&buf)), 10, 64)
check(err)

fmt.Println(v) // Hell
```

Почему мы вообще используем пакет `unsafe`, так это потому, что нет правила конверсии указателя на байт слайс в указатель на строку. Но есть правило конверсии указателя на слайс в `unsafe.Pointer` и правило конверсии `unsafe.Pointer` в указатель на строку. Как-то так:

```
(*string)(&buf)                 // Cannot convert an expression of the type '*[]byte' to the type '*string'
(*string)(unsafe.Pointer(&buf)) // This is fine 🔥
```

Сейчас вместо вызова функции рантайма осталось только две инструкции:

```
	MOVQ	main.buf+32(SP), AX
	MOVQ	AX, main.a.ptr+24(SP)
```

## Преобразование из строки в слайс байт

Если мы можем конвертировать `[]byte` в `string`, значит можем и обратно? Так ведь, ведь так, да? Выглядит что так, действуем по аналогии:

```
type socket struct {}

func (socket) ReadFrom(dst []byte) (n, error) {
	return copy(dst, []byte{70, 71, 72}), nil
}

func main() {
	s := socket{}

	buf := [3]byte{}
	for {
		n, err := s.ReadFrom(buf[:])
		check(err)

		v, err := strconv.ParseUint64(buf[:n], 10, 64)
		check(err)

		fmt.Print(v)
	}
}
```

И сказал им: идите по всему миру и проповедуйте спеку всей твари. А спеке чёрным по белому написано:

> Strings are immutable: once created, it is impossible to change the contents of a string.

И если проверить, то с первого взгляда, так оно и есть:


```go

	buf.WriteString("Go strings are immutable")

	out := buf.String()

	doit(out)

	fmt.Println(out) // Go strings are mutable
}

func doit(s string) {
	b := *(*[]byte)(unsafe.Pointer(&s))

	for i := 0; i < 7; i++ {
		b[15+i] = b[17+i]
	}

	b[len(b)-2] = ' '
	b[len(b)-1] = ' '
}
```

Хиханьки-хаханьки заканчивается когда мы вспонимаем, что байт слайсы изменяемый тип данных. Например, мы можем растянуть байт слайс до его емкости. Так стоп, а какая вообще ёмкость у получившегося слайса?

```go

	here = string([]byte(here))

	out := *(*[]byte)(unsafe.Pointer(&here))

	fmt.Printf("%d %d\n", len(out), cap(out)) // 6 96
```

Емкость равна 96, шести не равно, ответы 0, 6 и 42 пояснения не требуют, но 96?

## "Правильный способ"

На самом деле в Go есть "правильный" способ преобразовать строку в слайс и обратно, без аллокации и копирования нижележащего массива (и даже в его использовании можно накосячить [^4]).

Идея заключается в том, что мы конструируем заголовки (и строки и байт-слайса) самостоятельно и копируем поля из структуры из которой мы преобразовываем в только что созданную структуру.

```go
func btos(in []byte) (out string) {
	slice := (*reflect.SliceHeader)(unsafe.Pointer(&in))
	str := (*reflect.StringHeader)(unsafe.Pointer(&out))
	str.Data = slice.Data
	str.Len = slice.Len

	return s
}

func stob(in string) (out []byte) {
	slice := (*reflect.SliceHeader)(unsafe.Pointer(&out))
	str := (*reflect.StringHeader)(unsafe.Pointer(&in))
	slice.Data = str.Data
	slice.Len = str.Len
	slice.Cap = str.Len

	return s
}
```

Этот подход отличается концептуально. Мы больше не интерпретируем блок памяти от слайса как блок памяти строки. Мы создаём заголовок для слайса байт, а далее заполняем руками поля. Если компилятор будет изменять порядок следования полей или их смещения, то этот код продолжит работать.

Сложность и изначальное представление строк, как неизменяемых типов, приводит к тому что функциональности конверсии (без выделения памяти и копирования) нет в стандарной библиотеке Go, по крайней мере, это так со [слов разработчиков](https://github.com/golang/go/issues/25484#issuecomment-390846454).

## Заключение

Цель статьи не побудить использовать (даже правильно) пакет `unsafe`, а наоборот, показать с какими сложностями можно столкнуться при его использовании.

Так же проблема состоит в том, что язые Go, а именно компилятор языка, не подталкивает вас к правильному решению [^3], ему и с конверсией `*string -> unsafe.Pointer -> *[]byte` живётся хорошо. Для компилятора это одинаковые способы. Оба компилируются и работают. Корректную работу должен обеспечить программист. По моему мнению любой код содержащий `unsafe` должен содержать пояснение, что результат работы корректен и не противоречив.
